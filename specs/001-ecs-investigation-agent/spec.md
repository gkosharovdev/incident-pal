# Feature Specification: Production Investigation Agent

**Feature Branch**: `001-ecs-investigation-agent`  
**Created**: 2026-04-30  
**Status**: Draft  
**Input**: User description: "Build a production investigation agent for AWS ECS services. The agent helps engineers investigate user-impacting observations, starting with cases like 'a customer did not receive a notification email.' Given a service name, environment, entity type and entity identifier, and time window, it should inspect the codebase, query read-only production telemetry such as CloudWatch logs and ECS deployment metadata, correlate evidence across service boundaries, form and refine hypotheses, and produce a concise Markdown analysis report with timeline, evidence, likely failure point, confidence, and recommended next actions. The system must prioritize safety: read-only production access, auditable tool calls, no state-changing operations, explicit handling of uncertainty, and reproducible investigation traces. It should be extensible through narrow tools or MCP integrations for logs, deployments, service catalog, customer correlation, notification/outbox data, and email delivery status."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Investigate a User-Impacting Observation (Priority: P1)

An on-call engineer receives a report of a user-impacting observation — for example, a customer did not receive a notification, a payment was not processed, data appears incorrect, or an order is stuck in the wrong status. The engineer opens the investigation agent, provides the affected service name, environment (e.g., production), at least one linking key (e.g., entity type `order` + entity identifier `order-id-12345`, or an HTTP correlation ID), and an optional time window. The agent autonomously queries production telemetry — logs, deployment records, and any registered domain-specific data sources — across the relevant services, correlates evidence, and returns a Markdown report identifying the likely failure point, with a confidence level and recommended remediation actions.

**Why this priority**: This is the primary use case the system is built around. It delivers immediate, tangible value to engineers during incidents by reducing manual investigation time from hours to minutes, regardless of the observation type.

**Independent Test**: Can be fully tested by providing a known incident (where root cause is already established) as input and verifying the agent's report correctly identifies the failure point, timeline, and recommended actions.

**Acceptance Scenarios**:

1. **Given** a service name, environment, at least one linking key, and an optional observation description are provided, **When** the agent is invoked, **Then** it returns a Markdown report containing a timeline, evidence summary, likely failure point, confidence level, and recommended next actions within the configured wall-clock timeout (default: 10 minutes).
2. **Given** an observation description is provided (e.g., "payment not processed"), **When** the agent generates the report, **Then** the observation description appears verbatim in the report summary and the initial hypothesis reflects the described problem.
3. **Given** the agent queries production systems, **When** all tool calls are executed, **Then** every tool call is logged in an auditable trace that can be reviewed after the investigation.
4. **Given** the agent cannot find conclusive evidence, **When** it produces the report, **Then** it explicitly states uncertainty and lists what evidence was inspected and what gaps remain.

---

### User Story 2 - Cross-Service Evidence Correlation (Priority: P2)

An engineer is investigating an observation that spans multiple services — for example, a payment event was generated by one service, processed by another, and confirmed by a third; or a notification was queued by one service and dispatched by another; or an order status transition was triggered in one service but never reflected in another. The agent follows the evidence chain across service boundaries using whichever linking key the engineer provides: an entity identifier (e.g., order ID), an HTTP request correlation ID, or a Kafka message ID. Any single identifier is sufficient to start an investigation; the agent extracts and follows all associated identifiers it encounters as the investigation proceeds.

**Why this priority**: Most production incidents in distributed systems involve more than one service. Without cross-service correlation, the agent's value is limited to single-service investigations.

**Independent Test**: Can be tested independently by providing an entity identifier that is traceable across at least two known services in a staging or canary environment, and verifying the agent follows the trail across both.

**Acceptance Scenarios**:

1. **Given** an entity identifier that appears in logs across multiple services, **When** the agent investigates, **Then** the report includes evidence gathered from all services where the linking key or a derived linking key appears in logs within the time window.
2. **Given** a deployment occurred in one service during the time window, **When** the agent correlates evidence, **Then** the report links the deployment event to the observed impact and includes it in the timeline.

---

### User Story 3 - Reproducible Investigation Trace (Priority: P3)

An engineer who ran an investigation earlier wants to share the full investigation trace with a colleague or store it for post-incident review. The agent produces a self-contained, reproducible trace that records every data source queried, every tool call made, and every piece of evidence considered.

**Why this priority**: Reproducibility is critical for post-incident reviews and knowledge sharing, but it is not required for the core investigation flow.

**Independent Test**: Can be tested independently by running an investigation, saving the trace, and then verifying a colleague can read the trace and understand exactly what was queried and what conclusions were drawn.

**Acceptance Scenarios**:

1. **Given** an investigation is completed, **When** the trace is saved, **Then** it contains the full list of tool calls (with inputs and outputs), the hypotheses considered, and the final conclusions.
2. **Given** an investigation trace exists, **When** a colleague reads it, **Then** they can reproduce the same queries against the same time window and arrive at the same evidence.

---

### User Story 4 - Extend the Agent with a New Data Source (Priority: P4)

A platform engineer wants to add support for querying a new data source (e.g., an email delivery status API or a service catalog) without modifying the core agent logic. They register a new narrow tool or MCP integration, and the agent discovers and uses it in future investigations.

**Why this priority**: Extensibility ensures the agent grows with the platform's tooling without requiring core rewrites, but is secondary to delivering the core investigation value.

**Independent Test**: Can be tested by registering a mock tool that returns a known response and verifying the agent invokes it and incorporates its output into the investigation report.

**Acceptance Scenarios**:

1. **Given** a new tool is registered following the agent's extension contract, **When** the agent runs an investigation where that tool is relevant, **Then** it calls the tool and includes its output in the evidence.
2. **Given** a registered tool is unavailable or returns an error, **When** the agent encounters the failure, **Then** it records the failure in the trace, notes the gap in the report, and continues the investigation using available evidence.

---

### Edge Cases

- What happens when the provided time window is very broad (e.g., 7 days) and would return an overwhelming volume of log data? The agent must bound its queries and surface the most relevant evidence rather than failing or returning unstructured noise.
- What happens when the entity identifier does not appear in any queried data source? The agent must report "no evidence found" clearly, listing every source it checked, rather than returning a vacuous or misleading report.
- What happens when a required data source is unavailable (e.g., a log query service is down)? The agent must continue with available sources, note the unavailability in the trace and report, and qualify its confidence accordingly.
- What happens when two or more plausible failure points are identified? The agent must present each hypothesis with its supporting evidence and a relative confidence score rather than arbitrarily selecting one.
- What happens when the requested service does not exist in the service catalog? The agent must return a clear error identifying the unknown service before beginning the investigation.
- What happens when the agent is invoked with no time window? It must apply a default window of 1 hour (past 60 minutes from invocation time) and document this in the report.
- What happens when a log query is estimated to exceed the per-investigation scan budget? The agent must halt that specific query before execution, record the budget exhaustion in the trace, note the evidence gap in the report, and continue with remaining queries that fit within the budget.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST accept a service name, environment, and time window as required investigation inputs, plus at least one linking key from: entity identifier (with entity type), HTTP request correlation ID, or Kafka message ID. All three linking key types are first-class — any one is sufficient to start an investigation. The system MUST also accept an optional free-text `observationDescription` (e.g., "payment not processed", "order stuck in pending status", "customer data appears incorrect") that the agent uses to prime its initial hypothesis and focus evidence gathering. When provided, the observation description MUST be included verbatim in the report summary.
- **FR-002**: The system MUST query production telemetry sources (logs, deployment records) in a read-only manner — no writes, deletes, or state-changing operations are permitted at any point during an investigation.
- **FR-003**: The system MUST produce a Markdown investigation report containing: a chronological timeline of relevant events, evidence gathered per data source, the most likely failure point (defined as the point in the system where the observed discrepancy most likely originated — a pipeline stage, a specific service, or a data mutation — regardless of observation type), a confidence level for each conclusion, and recommended next actions.
- **FR-004**: The system MUST record an auditable trace of every tool call made during an investigation, including tool name, inputs, outputs, and timestamp.
- **FR-005**: The system MUST explicitly represent uncertainty — when evidence is insufficient or ambiguous, the report must state this rather than presenting an unsupported conclusion.
- **FR-006**: The system MUST correlate evidence across service boundaries using the provided linking key (entity identifier, HTTP request correlation ID, or Kafka message ID) as the starting point, and MUST automatically extract and follow additional linking keys of any type encountered in logs or traces during the investigation.
- **FR-007**: The system MUST apply a default time window of 1 hour (the past 60 minutes from invocation time) when none is provided, and MUST document this default explicitly in the report.
- **FR-008**: The system MUST gracefully handle unavailable or erroring data sources by continuing the investigation, recording the failure in the trace, and qualifying confidence in the report accordingly.
- **FR-009**: The system MUST support registration of additional narrow tools or integrations for new data sources without requiring changes to core investigation logic.
- **FR-010**: The system MUST validate that the specified service exists in the known service catalog before beginning an investigation, and return a clear error if not found.
- **FR-011**: The investigation mechanism is observation-type agnostic — the same agent, tool interface, and reporting structure apply regardless of whether the observation is a missing notification, a failed payment, faulty data, or an incorrect status. The system MUST ship with three built-in core tools covering cross-cutting evidence sources: CloudWatch log queries, ECS deployment metadata, and customer/entity correlation lookup. Additionally, a notification outbox tool MUST be bundled as a reference extension implementation — fully functional but registered as a domain-specific extension rather than a core tool — to serve as a concrete template for teams building payment, status, or other domain-specific tools. All other domain-specific tools (e.g., email delivery status, payment gateway status) are registered via the extension mechanism (FR-009).
- **FR-012**: The system MUST bound log queries to prevent runaway data retrieval — queries returning results above a configurable result-count threshold must be truncated and the truncation noted in the trace.
- **FR-013**: The system MUST assume log entries are structured JSON and extract linking keys (entity identifiers, HTTP correlation IDs, Kafka message IDs) by querying specific named fields. Log entries that are not valid JSON MUST be recorded in the trace as unparseable and skipped rather than causing the investigation to fail.
- **FR-014**: The system MUST enforce a configurable per-investigation data scan budget (measured in volume of data scanned). Before executing any log query estimated to exceed the remaining budget, the system MUST halt that query, record the budget exhaustion in the trace, and note the gap in the report rather than proceeding with an over-budget query.
- **FR-015**: The system MUST enforce a configurable wall-clock timeout per investigation (default: 10 minutes, configurable via `MAX_DURATION_MS` env var). When the timeout expires, the agent MUST halt the current tool call, mark the investigation status as `timed-out`, and return the partial report with a `## ⚠️ Investigation Timed Out` warning section as the first report section. The timeout is distinct from the per-investigation iteration limit (default: 20 tool calls, configurable via `MAX_ITERATIONS` env var) — whichever limit is reached first halts the investigation.

### Key Entities

- **Investigation**: A single invocation of the agent; defined by service name, environment, time window, at least one linking key (entity identifier, HTTP request correlation ID, or Kafka message ID), and an optional observation description. Status values: `running`, `complete`, `failed`, `budget-exhausted`, `timed-out`. Produces a report and a trace.
- **Linking Key**: An identifier used to correlate evidence across service boundaries. Supported types: entity identifier (e.g., order ID, customer ID), HTTP request correlation ID, Kafka message ID. All types are first-class inputs; any one is sufficient to initiate an investigation. Additional linking keys discovered during investigation are automatically followed.
- **Report**: The human-readable Markdown output of an investigation; contains timeline, evidence, hypotheses, confidence levels, and recommended actions. The `likelyFailurePoint` field describes the origin of the observed discrepancy — this may be a pipeline stage, a specific service boundary, or a data mutation point, depending on the observation type. The same structure applies for all observation types (notifications, payments, data quality, status).
- **Trace**: The machine-readable, append-only audit log of all tool calls, evidence collected, and hypotheses evaluated during an investigation; supports reproducibility.
- **Tool**: A narrow, read-only integration with a data source (e.g., log query, deployment lookup, email delivery status); the unit of extensibility.
- **Hypothesis**: A proposed explanation for the observed impact; associated with supporting evidence and a confidence score; multiple hypotheses may coexist in a single report.
- **Evidence Item**: A single data point retrieved from a tool call; associated with source, timestamp, and relevance to the investigation.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Engineers can complete a full investigation of a known incident (with ground truth established) and receive a correct root-cause identification in the report at least 80% of the time. The golden-set eval suite validating this criterion MUST contain at least 10 scenarios spanning at least 3 distinct observation types (e.g., notification failure, payment failure, incorrect status) — accuracy measured against a notification-only scenario set does not satisfy this criterion.
- **SC-002**: An engineer using the agent produces a complete or partial investigation report within 10 minutes — guaranteed by the wall-clock timeout (FR-015). For incidents that previously required 30+ minutes of manual log trawling, the agent delivers a report (complete or partial with a gap summary) within that window 100% of the time.
- **SC-003**: 100% of tool calls executed during any investigation are captured in the auditable trace — no tool call is unrecorded.
- **SC-004**: The agent correctly handles unavailable data sources (continues investigation, documents gap) in 100% of cases — it never silently drops evidence gaps.
- **SC-005**: A new data source integration (tool) can be registered and used in an investigation without changing the core agent, measurable by a new tool being callable in under 1 day of integration work.
- **SC-006**: Investigation reports produced for the same inputs (same service, entity, time window) against the same underlying data are reproducible — a second run returns the same timeline and evidence items.

## Clarifications

### Session 2026-04-30

- Q: What identifier types should be supported as first-class cross-service linking keys — entity ID only, or also HTTP correlation IDs and Kafka message IDs? → A: All three (entity identifier, HTTP request correlation ID, Kafka message ID) are first-class inputs; any one is sufficient to start an investigation.
- Q: How should the agent handle PII (customer emails, order details) present in logs and investigation traces? → A: No special handling by the agent — PII protection is entirely delegated to the underlying data source access controls.
- Q: What log format should the agent assume when extracting linking keys from log entries? → A: Structured JSON logs; the agent queries specific named fields to extract entity identifiers, HTTP correlation IDs, and Kafka message IDs.
- Q: How should the agent handle CloudWatch query cost and rate limits? → A: Enforce a configurable per-investigation scan budget (data volume limit); halt and report if the budget would be exceeded before executing the query.
- Q: What should the default time window be when none is provided by the engineer? → A: 1 hour (the past 60 minutes from the time of invocation).
- Q: Is the investigation scope limited to missing notifications, or should it cover any user-impacting observation? → A: The scope is open-ended — missing notifications, failed payments, faulty data, incorrect status, and any other user-impacting observation are all valid. The agent mechanism is observation-type agnostic; only the registered tools are domain-specific.
- Q: Should the built-in toolset ship domain-specific tools (notification outbox, email delivery) as core, or keep the core generic and move domain tools to extensions? → A: Three cross-cutting tools built-in (CloudWatch, ECS, entity correlation); notification outbox bundled as a reference extension implementation; all other domain-specific tools (email delivery, payment gateway, status) are pure extensions.
- Q: Should the investigation input include a way for engineers to describe the observed problem? → A: Yes — add an optional free-text `observationDescription` (e.g., "payment not processed", "order stuck in pending status") used by the agent to prime its initial hypothesis; included verbatim in the report summary.
- Q: What happens if an investigation exceeds 10 minutes? → A: Enforce a hard wall-clock timeout (default 10 min, configurable); on expiry halt the agent, set status to `timed-out`, and return a partial report with a prominent warning. SC-002 is now a guarantee, not a target.
- Q: Should the golden-set eval suite cover multiple observation types to validate SC-001 across the full investigation scope? → A: Yes — the ≥10 scenario suite must span at least 3 distinct observation types (e.g., notification failure, payment failure, incorrect status); notification-only evals do not satisfy SC-001.
- Q: Should the report's `likelyFailurePoint` field be broadened beyond pipeline stages to cover data mutations and status inconsistencies? → A: Yes — broaden definition to "the point in the system where the observed discrepancy most likely originated"; same single field covers all observation types.

## Assumptions

- Engineers running investigations are authenticated members of the on-call or engineering team; access control to production data sources is enforced by the underlying data source systems, not by the agent itself. The agent does not perform PII filtering, redaction, or anonymisation — data sensitivity is governed entirely by the access policies of each data source.
- All production data sources support read-only query access; the agent does not need to manage credentials for individual sources beyond what the integration layer provides.
- The service catalog is the authoritative source for valid service names and environments; the agent uses it for input validation.
- The investigation scope is open-ended — any user-impacting observation (missing notification, failed payment, faulty data, incorrect status, etc.) is a valid investigation target. The initial built-in toolset covers common cross-cutting sources (CloudWatch logs, ECS deployments, entity correlation); domain-specific tools are added via the extension mechanism as investigation use cases expand.
- Mobile or web UI is out of scope for v1; the agent is invoked programmatically (CLI, API, or AI agent harness) and produces Markdown output.
- The agent is single-tenant per invocation; it does not need to support concurrent investigations sharing state.
- Log retention and data source availability are managed by the platform; the agent assumes data exists within the requested time window and gracefully handles cases where it does not.
- All services emit structured JSON logs; the agent does not support unstructured or plain-text log formats. Services that do not emit JSON logs are outside the scope of log-based evidence gathering.
- The agent does not need to send alerts, open tickets, or take any remediation action — its output is informational only.
