# Implementation Plan: Aurora PostgreSQL DB Query Tool

**Branch**: `002-aurora-db-query-tool` | **Date**: 2026-05-01 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/002-aurora-db-query-tool/spec.md`

## Summary

Implement `AuroraDbTool`, a read-only extension tool that queries Aurora PostgreSQL databases for services under investigation, enabling the agent to correlate log evidence with the actual database state in a specific environment. The tool resolves connection details from the service catalog, authenticates via IAM (or env-var fallback), enforces SELECT-only execution, and returns structured row data with scan budget tracking — following the identical extension pattern established by `NotificationOutboxTool`.

## Technical Context

**Language/Version**: TypeScript 5.8 (strict mode), Node.js ≥ 22  
**Primary Dependencies**: `pg` v8 (new), `@aws-sdk/rds-signer` (new), `js-yaml` (existing), `zod` (existing)  
**Storage**: Aurora PostgreSQL (read-only query target — no local state)  
**Testing**: Vitest (existing)  
**Target Platform**: Linux server (same as all other tools)  
**Project Type**: Extension tool library — registered at runtime by the caller  
**Performance Goals**: Query results available within 10 s per-query timeout (default); tool must not block the agent's ReAct loop  
**Constraints**: SELECT-only, no writes, cyclomatic complexity ≤ 10 per function, zero `any` without suppression comment  
**Scale/Scope**: One connection per `invoke()` call; no connection pooling required (per spec Assumptions)

## Implementation Phases

### Phase 1: AuroraDbGuard + AuroraDbCatalogReader

Two pure, stateless modules with no external I/O — unit-testable in isolation.

**`AuroraDbGuard.ts`**

```typescript
// Exported function — all logic fits in ≤ 5 lines, CC = 2
export function assertSelectOnly(query: string): void | never {
  const firstToken = query.trim().split(/\s+/)[0]?.toUpperCase() ?? "";
  if (firstToken !== "SELECT") {
    throw new Error(`WRITE_REJECTED: Only SELECT statements are permitted. Got: ${firstToken}`);
  }
}
```

**`AuroraDbCatalogReader.ts`**

- Reads `service-catalog.yml` via `readFileSync` + `js-yaml` (same pattern as `ServiceCatalogTool`)
- Exposes one method: `resolve(serviceId: string, environment: string): AuroraDatabaseConfig | null`
- `AuroraDatabaseConfig` type defined here (not in `src/models/`)
- Catalog entry shape: see `data-model.md`
- No dependency on `ServiceCatalogTool` — reads the same file independently to respect Interface Segregation

### Phase 2: AuroraDbCredentials

**`AuroraDbCredentials.ts`**

- One exported async function: `resolvePassword(config: AuroraDatabaseConfig): Promise<string>`
- Branch on `credentialSource`:
  - `"iam"` → call `RDSSigner.getAuthToken({ hostname, port, region, username })`
  - `"env-var"` → read `process.env[config.envPasswordVar]`; throw if missing
- Cyclomatic complexity: 3 (two branches + error path)

---

### Phase 3: AuroraDbTool

**`AuroraDbTool.ts`**

Constructor:
```typescript
constructor(
  catalogPath: string,
  options?: { maxRows?: number; queryTimeoutMs?: number }
)
```

`invoke(input: unknown): Promise<ToolResult>` flow (see data-model.md state transitions):

1. Validate input with Zod schema (same pattern as other tools)
2. Call `assertSelectOnly(input.query)` — return error result on throw
3. Call `catalogReader.resolve(serviceId, environment)` — return error result if null
4. Call `resolvePassword(config)` — return error result on throw
5. Create `pg.Client`, connect with SSL `require`, run `SET statement_timeout = ${timeoutMs}`
6. Execute SELECT, collect up to `maxRows + 1` rows to detect truncation
7. Close client (in `finally` block)
8. Return `{ success: true, data: AuroraDbResult, scanBytesUsed: rowCount * 1024, truncated }`

Each of steps 2–8 is a single logical responsibility; helper methods keep `invoke()` CC ≤ 10.

**INPUT_SCHEMA** (JSONSchema7):
```typescript
{
  type: "object",
  properties: {
    serviceId: { type: "string", minLength: 1 },
    environment: { type: "string", enum: ["production", "staging", "canary"] },
    query: { type: "string", minLength: 1, description: "SELECT statement to execute" },
    maxRows: { type: "number", description: "Override default row cap (optional)" },
  },
  required: ["serviceId", "environment", "query"],
  additionalProperties: false,
}
```

### Phase 4: Tests

**Unit tests** (`tests/unit/tools/aurora-db.test.ts`):

| Test case | What it asserts |
|---|---|
| Valid SELECT returns rows | `success: true`, rows array populated |
| Non-SELECT rejected before connect | `success: false`, error contains `WRITE_REJECTED`, `pg.Client` never instantiated |
| No catalog entry → structured error | `success: false`, error contains `NO_DB_CONFIGURED` |
| Row cap enforced | `truncated: true` when DB returns maxRows+1 rows |
| DB unreachable → structured error | `success: false`, error propagated, no throw |
| Query timeout → structured error | `success: false`, error contains timeout signal |
| `scanBytesUsed` equals `rowCount * 1024` | Budget accounting is correct |

All `pg.Client` interactions are mocked via `vi.mock` or constructor injection.

**Integration test** (`tests/integration/extensions/aurora-db.test.ts`):

- Registers `AuroraDbTool` in `ToolRegistry`
- Asserts tool is discoverable by name `"aurora-db"`
- Asserts tool definition appears in `getToolDefinitions()` output (LLM-facing schema)
- Asserts error result (not throw) when catalog has no entry

Uses a mock `pg.Client` via dependency injection — no live Aurora cluster required.
