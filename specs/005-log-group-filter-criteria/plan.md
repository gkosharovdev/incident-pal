# Implementation Plan: Log Group Filter Criteria

**Branch**: `005-log-group-filter-criteria` | **Date**: 2026-05-05 | **Spec**: [spec.md](spec.md)

## Summary

Replace the single fixed `logGroup` field in the service catalog with a `logGroupFilters` list so the agent can discover all relevant CloudWatch log groups at investigation start — covering ECS tasks, RDS Proxy, ALB, API Gateway, Lambda, and any other layer that writes to CloudWatch. A new `LogGroupDiscoveryTool` resolves filter expressions to concrete group names via `DescribeLogGroups`. Legacy single-group catalog entries are fully backward-compatible.

## Technical Context

**Language/Version**: TypeScript 5.x, Node.js 22 LTS (strict mode)
**Primary Dependencies**: `@aws-sdk/client-cloudwatch-logs` (already installed — `DescribeLogGroupsCommand` already imported in `CloudWatchLogsTool.ts`)
**Storage**: YAML service catalog file (read-only at runtime)
**Testing**: Vitest — unit tests with recorded fixtures; existing test patterns in `tests/unit/tools/`
**Target Platform**: Node.js CLI / TUI (same as existing tools)
**Project Type**: CLI/TUI investigation agent
**Performance Goals**: Discovery of up to 50 log groups in ≤5 seconds (single DescribeLogGroups API call covers the default cap)
**Constraints**: Read-only AWS access; no new SDK dependencies; backward-compatible with all existing catalog entries
**Scale/Scope**: Typically 2–10 log groups per service; hard cap at 50

## Phase 0: Research

Complete — see [research.md](research.md). All decisions resolved:

- AWS API: `DescribeLogGroupsCommand` with `logGroupNamePrefix` / `logGroupNamePattern`
- Architecture: new `LogGroupDiscoveryTool`; agent calls catalog then discovery then cloudwatch
- Multi-group queries: per-group calls (batching deferred)
- Backward compat: synthesise prefix filter from legacy `logGroups` field
- Cap: 50 groups default, configurable via `maxLogGroups` catalog field

## Phase 1: Design & Contracts

Complete — see [data-model.md](data-model.md) and [contracts/](contracts/).

### Key design decisions

**`ServiceCatalogTool` changes** (open for extension, closed for modification of return shape):
- `ServiceEntry` interface gains optional `logGroupFilters: Record<string, LogGroupFilter[]>` and `maxLogGroups?: number`
- `ServiceLookupResult` gains `logGroupFilters: LogGroupFilter[]` and `maxLogGroups: number`
- `logGroup` (existing) is retained; backward-compat synthesis happens inside the tool
- Validation at load time: invalid filter `type` values → throw with clear message

**`LogGroupDiscoveryTool`** (new, follows Tool interface):
- Input: `{ filters: LogGroupFilter[], maxGroups?: number }`
- Calls `DescribeLogGroups` once per filter type (prefix vs. pattern), paginates until cap
- De-duplicates by log group name; attributes each to its first matching filter
- Returns `{ groups, capped, totalFound }` on success
- Fail-fast on any AWS error (does not return partial results)
- Registered in `useInvestigation.ts` alongside existing tools

**`prompts.ts` additive update**:
Append to step 1 of the Investigation Process section:
```
1b. If the service catalog result contains `logGroupFilters`, call `log-group-discovery` with those filters to obtain the concrete list of log group names before issuing any CloudWatch queries. Proceed with all discovered groups. If discovery returns no groups for a filter, note it and continue.
```

**`service-catalog.yml`** update:
Add `logGroupFilters` to `booking-service` so it can discover ECS, RDS Proxy, ALB, and other layers dynamically.

### Wiring

`LogGroupDiscoveryTool` is injected in `src/tui/hooks/useInvestigation.ts` (the TUI wiring) and `src/cli/index.ts` (the headless CLI wiring), alongside the existing three tools. Both receive the shared `CloudWatchLogsClient` instance.
