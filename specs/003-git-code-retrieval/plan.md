# Implementation Plan: Git Code Retrieval & Repo Documentation Tools

**Branch**: `003-git-code-retrieval` | **Date**: 2026-05-01 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/003-git-code-retrieval/spec.md`

## Summary

Implement two focused, read-only extension tools that close the incident analysis loop by correlating deployed code with observed failures. `GitCodeRetrievalTool` provides four operations against a git repository at a specific commit: commit metadata + diff, file content retrieval, version comparison, and symbol search. `RepoDocumentationTool` discovers and returns architecture and business documentation (README, AGENTS.md, ADRs, .specify files) from the repository HEAD. Both tools share a `GitProvider` interface and a `GitCatalogReader` helper that resolves the repository URL from the service catalog. The agent's system prompt is updated to guide synthesis of all findings into a "Code & Architecture Analysis" report section.

## Implementation Phases

### Phase 1: Shared infrastructure вЂ” GitProvider, GitHubProvider, GitCatalogReader

Three pure modules with no tool-specific logic вЂ” unit-testable in isolation.

**`GitProvider.ts`** вЂ” interface and return types only; no runtime dependencies

```typescript
export interface CommitInfo {
  sha: string;
  shortSha: string;
  message: string;
  authorName: string;
  authorEmail: string;
  timestamp: string;        // ISO 8601
  parentShas: string[];
  files: FileDiff[];
}

export interface FileDiff {
  filePath: string;
  previousPath: string | null;  // non-null for renames
  changeType: "added" | "modified" | "deleted" | "renamed";
  unifiedDiff: string;
  linesAdded: number;
  linesRemoved: number;
}

export interface FileContent {
  filePath: string;
  sha: string;
  content: string;
  sizeBytes: number;
}

export interface VersionComparison {
  baseSha: string;
  headSha: string;
  files: FileDiff[];
  totalFilesChanged: number;
  totalLinesAdded: number;
  totalLinesRemoved: number;
}

export interface SymbolMatch {
  filePath: string;
  lineNumber: number;
  matchContext: string;  // the line of code containing the match
}

export interface DirectoryEntry {
  path: string;
  type: "file" | "dir";
}

export interface GitProvider {
  getCommit(owner: string, repo: string, ref: string): Promise<CommitInfo>;
  getFileContent(owner: string, repo: string, ref: string, filePath: string): Promise<FileContent>;
  compareCommits(owner: string, repo: string, base: string, head: string): Promise<VersionComparison>;
  searchSymbol(owner: string, repo: string, symbol: string): Promise<SymbolMatch[]>;
  listDirectory(owner: string, repo: string, ref: string, dirPath: string): Promise<DirectoryEntry[]>;
}
```

**`GitHubProvider.ts`** вЂ” `Octokit` injected at construction time (testability)

- `getCommit`: `GET /repos/{owner}/{repo}/commits/{ref}` вЂ” parses commit + files array
- `getFileContent`: `GET /repos/{owner}/{repo}/contents/{path}?ref={ref}` вЂ” base64-decodes content
- `compareCommits`: `GET /repos/{owner}/{repo}/compare/{base}...{head}` вЂ” maps file list
- `searchSymbol`: `GET /search/code?q={symbol}+repo:{owner}/{repo}` вЂ” returns up to `MAX_SYMBOL_RESULTS` matches; note: GitHub code search indexes the default branch, not a specific ref (documented limitation in `quickstart.md`)
- `listDirectory`: `GET /repos/{owner}/{repo}/contents/{path}?ref={ref}` вЂ” when response is an array, maps entries

All methods throw a typed `GitProviderError` (extends `Error`) with a `statusCode` field on API errors.

### Phase 2: GitCodeRetrievalTool

**`GitCodeRetrievalTool.ts`**

Constructor:
```typescript
constructor(
  provider: GitProvider,
  catalogPath: string,
  options?: { maxDiffBytes?: number; maxSymbolResults?: number; requestTimeoutMs?: number }
)
```

`requestTimeoutMs` (default: 10 000 ms) is passed through to each `GitProvider` call. This enforces SC-002 (diff retrieval в‰¤ 5 s) with a safety margin. On timeout the provider throws a `GitProviderError` which `invoke()` catches and returns as a structured error.

Input schema (flat with `operation` discriminator):
```typescript
INPUT_SCHEMA = {
  type: "object",
  properties: {
    operation: {
      type: "string",
      enum: ["get-commit", "get-file", "compare", "search-symbol"],
      description: "Which retrieval operation to perform"
    },
    serviceId: { type: "string", description: "Service ID from the service catalog" },
    ref:       { type: "string", description: "Commit SHA, tag, or branch вЂ” required for get-commit, get-file, search-symbol" },
    filePath:  { type: "string", description: "File path within the repo вЂ” required for get-file" },
    baseRef:   { type: "string", description: "Base commit/tag вЂ” required for compare" },
    headRef:   { type: "string", description: "Head commit/tag вЂ” required for compare" },
    symbol:    { type: "string", description: "Method, class, or identifier to search for вЂ” required for search-symbol" },
  },
  required: ["operation", "serviceId"],
  additionalProperties: false,
}
```

`invoke()` flow:
1. Validate input fields for the given operation; return structured error for missing required fields
2. Call `catalogReader.resolve(serviceId)` вЂ” return error if no repository configured
3. Dispatch to private method: `handleGetCommit`, `handleGetFile`, `handleCompare`, `handleSearchSymbol`
4. Each handler calls the appropriate `GitProvider` method, applies payload cap, sets `truncated` if needed
5. Wrap caught `GitProviderError` into `{ success: false, data: null, error: message }`

`handleGetCommit` applies `maxDiffBytes` cap: sum the `unifiedDiff` lengths across all files; if exceeded, trim the last file diffs with a note and set `truncated: true`.

`handleCompare` calls `provider.compareCommits(owner, repo, baseRef, headRef)`. After receiving the result, it checks that `headSha` is a descendant of `baseSha` by comparing timestamps or commit graph order. If the base is newer than the head (reversed order), it returns `{ success: false, data: null, error: "INVALID_REF_ORDER: baseRef must be an ancestor of headRef" }` before applying the diff cap. This satisfies US3 acceptance scenario 2.

`handleSearchSymbol` caps results at `maxSymbolResults` (default: 50) and sets `truncated: true` if the GitHub API response indicates more results exist.

**`RepoDocumentationTool.ts`**

Constructor:
```typescript
constructor(
  provider: GitProvider,
  catalogPath: string,
  options?: { maxFileSizeBytes?: number; requestTimeoutMs?: number }
)
```

Well-known documentation paths scanned (in order):
```typescript
const WELL_KNOWN_DOC_PATHS = [
  "README.md",
  "AGENTS.md",
  ".specify",    // directory вЂ” listed then files fetched
  "adr",         // directory вЂ” listed then files fetched
  "docs/adr",    // directory вЂ” listed then files fetched
] as const;
```

Input schema:
```typescript
INPUT_SCHEMA = {
  type: "object",
  properties: {
    serviceId: { type: "string", description: "Service ID from the service catalog" },
  },
  required: ["serviceId"],
  additionalProperties: false,
}
```

`invoke()` flow:
1. Call `catalogReader.resolve(serviceId)` вЂ” return error if no repository configured
2. For each well-known path: try `provider.getFileContent` (for files) or `provider.listDirectory` + fetch (for directories). Silently skip any path that returns a 404.
3. For each successfully fetched file: apply `maxFileSizeBytes` cap, set `truncated: true` per file if content was trimmed
4. Return `{ success: true, data: DocumentationResult }` containing all collected files
5. Wrap caught `GitProviderError` into `{ success: false, data: null, error: message }`

Reads from the default branch HEAD (no ref input) вЂ” appropriate for documentation that tracks the main line of development.

### Phase 4: System prompt update

**`src/agent/prompts.ts`** вЂ” additive change only; existing `SYSTEM_PROMPT` constant gains a new section:

Append to `SYSTEM_PROMPT` after the existing "Observation Types" section:

```
## Code & Architecture Analysis

When `git-code-retrieval` and `repo-documentation` tools are available, follow this investigation sequence for deployments detected within the time window:

1. **Architecture context first**: Call `repo-documentation` to read README, AGENTS.md, ADR files, and .specify documentation. Build a mental model of the service's design and business flow before inspecting code changes.
2. **Code changes**: Call `git-code-retrieval` with `operation: "get-commit"` (or `"compare"` for multi-commit deployments) to retrieve the diff introduced by the deployment.
3. **Code flow tracing**: For each suspicious change, use `operation: "search-symbol"` to locate the method or class referenced in the logs, then `operation: "get-file"` to read the full file and trace the invocation chain.
4. **DB correlation**: Cross-reference the code path with the current entity state retrieved from `aurora-db`. Look for mismatches between what the code intends and what the database contains.
5. **Synthesis**: When producing the report, include a "Code & Architecture Analysis" section that summarises: the architectural context, the specific code change most likely responsible, the call chain from entry point to failure, and how the current DB state corroborates or contradicts the hypothesis.
```

`SYSTEM_PROMPT` is a static string constant вЂ” this additive change preserves prompt cache validity (the cache key covers the full string; any change invalidates the cache for one warm-up request, after which it re-establishes).
